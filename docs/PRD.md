# DesignX 产品需求文档（PRD）

## 1. 文档信息

- 产品：DesignX
- 版本：MVP
- 状态：需求确认中
- 目标平台：企业内网 Windows 10/11
- 产品形态：单用户 Electron 桌面应用

## 2. 产品定位

DesignX 是面向系统设计人员和模块负责人的本地研发治理工作台。它将代码仓中的实际变更与已确认的业务需求、系统设计和编程规范进行对照，输出同时具备代码证据和知识证据的发现。

DesignX 不替代代码评审，也不自动修改代码。它解决的是一个更具体的问题：当代码持续变化时，负责人能否快速判断“这次变更是否偏离了已经确认的约束，以及依据是什么”。

## 3. 目标用户与使用场景

### 3.1 目标用户

- 系统设计人员
- 模块负责人
- 需要维护需求、设计与实现一致性的高级研发人员

MVP 不提供团队协作、用户体系和权限管理。

### 3.2 核心场景

1. 用户接入一个企业内部 Git 仓库。
2. 用户导入需求、设计或编程规范，并发布为可引用的知识版本。
3. 用户选择代码仓、分支和知识版本发起分析。
4. DesignX 根据代码变更与影响范围生成发现。
5. 用户从发现中同时查看代码位置、知识依据、判断说明和修改建议。

## 4. 产品目标

MVP 必须形成以下最短闭环：

```text
接入代码仓 → 发布知识版本 → 发起一致性分析 → 查看可追溯发现
```

闭环成立的判断标准：

- 用户不离开桌面应用即可完成上述流程。
- 每条正式发现可以定位到代码文件与 Commit。
- 需求、设计和规范类发现可以定位到所使用的知识版本与原文位置。
- 分析失败不会覆盖上一次成功结果，用户可以重新运行。

## 5. MVP 范围

### 5.1 包含

- 单用户、单机、企业内网运行。
- 通过 Git 地址添加代码仓，管理默认分支并手动同步。
- 导入 Markdown、PDF、DOCX 文档。
- 将文档整理为包含 `SKILL.md`、`references/` 和来源文件的版本化知识包。
- 在发布前预览和修改知识草稿；已发布版本不可原地修改。
- 选择代码仓、知识版本和分析范围发起一致性分析。
- 调用 OpenAI-compatible 的企业内部模型 API。
- 查看分析任务状态与失败原因。
- 查看、筛选和忽略发现。
- 在同一详情界面查看代码证据、知识证据、判断说明和修改建议。
- 业务数据保存在用户选择的本地工作区。

### 5.2 不包含

- 健康度评分、趋势图、周期报告和报告导出。
- 后台轮询、定时分析、系统托盘和开机启动。
- MR/PR 评论、合并门禁、分支保护和自动整改。
- 用户、角色、SSO、权限、多租户和团队协作。
- 审批、责任人、豁免流程和通知渠道。
- RAG、Embedding、向量数据库和独立搜索服务。
- 自定义规则 DSL、自定义提示词编排和自定义报告模板。

## 6. 核心概念

### 6.1 代码仓

一个代码仓对应一个 Git 远程地址、一个默认分支和一个本地副本。MVP 只分析默认分支。

### 6.2 知识包

知识包是可被分析任务引用的版本化约束集合，类型包括：

- 业务需求
- 系统设计
- 编程规范

草稿允许修改；发布后生成不可变版本。历史分析始终引用运行时使用的具体版本。

### 6.3 分析任务

一次分析任务固定以下输入：

- 代码仓与分支
- 起止 Commit 或当前基线
- 一个或多个已发布知识版本
- 用户补充的检查范围

任务完成后保存分析范围、实际加载的知识、发现和失败信息。

### 6.4 发现

发现是 DesignX 的核心输出。正式发现包含：

- 类型：代码质量、编程规范、业务需求一致性或系统设计一致性
- 严重度与置信度
- 文件、行号、符号和 Commit
- 知识包、版本、来源章节或 reference 路径
- 判断说明与修改建议

需求、设计和规范类结果缺少任一侧证据时，只能标记为“证据不足”，不能作为正式发现。

## 7. 功能需求

### 7.1 首次使用与工作区

- 首次启动时，用户选择一个本地目录作为 DesignX 工作区。
- 应用记住最近使用的工作区。
- 用户可在设置中更换工作区和查看数据目录。
- 工作区不可用时，应用说明原因并允许重新选择，不创建半完成状态。

### 7.2 代码仓

- 用户输入仓库名称、Git 地址和默认分支添加代码仓。
- 添加前验证 Git 可用性、地址访问能力和目标目录冲突。
- 添加成功后克隆代码仓。
- 用户可手动同步仓库。
- 页面展示当前 Commit、最近同步时间和 Git 状态。
- 同步失败时保留原有可用版本，并展示可执行的错误信息。

### 7.3 知识库

- 用户可一次导入一个或多个 Markdown、PDF、DOCX 文件。
- 应用保存原文件，并提取标题、章节和正文位置。
- 应用生成知识草稿，用户可以检查名称、类型、适用范围、`SKILL.md` 和 references。
- 发布前校验引用目标、来源映射和必要字段。
- 已发布知识只能创建新版本，不能覆盖修改。
- 知识列表展示名称、类型、版本、状态和更新时间。

### 7.4 新建分析

- 用户从首页或“新建分析”进入分析输入区。
- 用户必须选择一个可用代码仓和至少一个已发布知识版本。
- 默认分析范围为“上次成功分析 Commit 至当前 Commit”；没有基线时分析当前代码基线。
- 用户可用自然语言补充模块、目录、变更或关注点。
- 开始前明确展示代码仓、分支、知识版本和分析范围。
- 同一代码仓存在运行中任务时，阻止重复启动并跳转到现有任务。

### 7.5 分析执行

- 应用根据 Git Diff、变更文件及相关代码上下文确定分析范围。
- 应用根据文件路径、语言、模块和分析类型选择相关知识，不一次加载全部内容。
- 大模型输出必须通过结构化校验后才能生成发现。
- 任务状态包括：排队中、分析中、已完成、部分失败、失败。
- 用户可查看当前阶段和失败原因，并重新运行失败任务。
- 只有任务成功后才更新该仓库的分析基线。

### 7.6 分析任务

- 任务列表按开始时间倒序展示。
- 每行展示代码仓、Commit 范围、知识版本、状态、发现数量和耗时。
- 用户可打开任务查看输入范围、执行阶段、实际加载的知识和发现。
- 失败任务保留已生成的诊断信息，不显示为空白任务。

### 7.7 发现

- 发现列表支持按代码仓、任务、类型、严重度和状态筛选。
- 默认优先展示严重度高且置信度高的待处理发现。
- 用户可将发现标记为“已忽略”，并填写可选原因。
- 详情采用代码证据与知识证据并列布局。
- 代码证据展示文件路径、行号、Commit 和相关代码片段。
- 知识证据展示知识包、版本、章节、原文片段和来源文件。
- 判断说明必须区分已知事实与模型推断。

### 7.8 设置

设置使用弹窗，不建设独立页面。MVP 设置包括：

- 工作区目录
- 企业模型 API 地址与模型名称
- API 凭据
- 请求超时

凭据不得以明文写入工作区文件。

## 8. 信息架构与界面设计

### 8.1 应用结构

应用采用与 OpenAI Codex 桌面端相近的原生桌面结构：

- Windows 原生标题栏与菜单。
- 左侧固定导航栏。
- 右侧单一主工作区。
- 首页底部使用浮动分析输入框，作为主要操作入口。
- 次要配置通过弹窗或抽屉完成。

左侧导航固定为：

1. 新建分析
2. 代码仓
3. 知识库
4. 分析任务
5. 发现

不设置数据大屏和独立首页仪表盘。

### 8.2 首页

- 中央说明当前主要动作：“今天想检查哪个代码变更？”
- 提供“同步代码仓”“导入规范文档”“运行一致性分析”三个快捷入口。
- 底部分析输入框展示代码仓、知识版本、分支、分析模式和模型。
- 当缺少代码仓或知识版本时，输入框直接引导用户完成缺失步骤。

### 8.3 视觉原则

- 主工作区使用纯白背景，左侧栏使用低饱和冷灰蓝。
- 以留白、细边框和紧凑文字层级组织信息，不使用卡片堆叠。
- 主要强调色仅用于当前选择、进行中状态和主操作。
- 图标使用统一的单色线性风格。
- 中文界面优先使用系统 UI 字体，控件保持 Windows 桌面应用密度。

## 9. 状态与反馈

- 空状态必须说明下一步可执行动作。
- 长任务展示当前阶段，不只显示无含义的加载动画。
- 成功操作在当前上下文内反馈，不依赖全局消息中心。
- 错误信息说明失败对象、失败阶段和建议动作。
- 删除、忽略、切换工作区等会改变结果的操作需要确认。
- 应用重启后可恢复代码仓、知识、任务和发现状态。

## 10. 安全与数据边界

- 代码、文档、知识包、任务和发现只保存在用户选择的本地工作区。
- 模型请求只允许发送到用户配置的企业内部 API 地址。
- 日志记录任务、阶段、时间和错误，不默认复制完整代码与原文。
- Git 与模型凭据使用 Windows Credential Manager 或 DPAPI 保存。
- 渲染进程不得直接访问 Node.js、文件系统或凭据。

## 11. 产品约束

- 桌面容器使用 Electron。
- 界面使用 React、TypeScript 和 Vite。
- Git 操作使用 Windows 系统 Git。
- 不依赖 Docker、外部数据库或后台服务。
- 模型接口兼容 OpenAI API 格式。

## 12. MVP 验收标准

1. 用户可在全新 Windows 环境中选择工作区并进入应用。
2. 用户可添加并同步一个企业 Git 仓库，界面显示当前 Commit 和 Git 状态。
3. 用户可导入支持的文档，检查草稿并发布不可变知识版本。
4. 用户可选择代码仓与知识版本发起分析，并看到明确的执行状态。
5. 成功任务至少生成一条可同时追溯到代码位置和知识来源的正式发现，或明确说明未发现问题。
6. 用户可查看发现详情并将其标记为已忽略。
7. Git、文档解析或模型调用失败时，应用保留已有数据并允许重试。
8. 应用重启后，工作区中的代码仓、知识、任务和发现仍可恢复。
9. 主界面、导航、输入区和关键弹窗在 1280×720 及以上窗口中无内容遮挡或横向溢出。
